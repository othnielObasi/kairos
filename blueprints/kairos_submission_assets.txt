# KAIROS — Complete Submission Assets

## PROJECT TITLE
Kairos — The Governed Autonomous Trading Agent

## SHORT DESCRIPTION (≤ 160 characters)
Kairos is an autonomous trading agent that pays for its own governance. Every decision earns its execution through Circle Nanopayments on Arc.

## LONG DESCRIPTION

The internet made information programmable. Circle Nanopayments and Arc make value programmable. Kairos is what happens when you combine both.

Kairos is a production-grade autonomous trading agent that governs itself through payment. Every governance stage evaluation, every LLM reasoning call, every data API query, and every trade settlement is a real USDC transaction on Arc — paid by the agent, settled in real time, verifiable on-chain.

**The core primitive: agent pays for its own governance.**

Before any trade executes, Kairos must pass five governance stages: mandate enforcement, oracle integrity, execution simulation, supervisory decision, and risk routing. Each stage fires a Circle Nanopayment ($0.001 USDC) via the NOVTIA Runtime Governor. The agent cannot proceed without paying. This creates an economic feedback loop where every governed action has a cost — and that cost is the proof of governance.

**Four tracks, one system, one cycle.**

Track 1 — Agent-to-Agent Loop: Five Circle Nanopayments per cycle fire between the trading agent and the NOVTIA Governor. Machine-to-machine commerce, no human intermediary, instant confirmation, Arc batch settlement.

Track 2 — Per-API Monetisation: Five data feed calls via x402 through Circle Gateway, paying AIsa's 79-endpoint catalog per request. CoinGecko price data → $0.024. Twitter crypto sentiment → $0.0022. Financial news → $0.048. Perplexity market analysis → $0.012. No subscriptions. No API keys. Pay per call.

Track 3 — Usage-Based Compute: Every LLM inference call (Claude → Gemini → OpenAI failover chain) and every SAGE adaptive reflection cycle fires a Circle Nanopayment. Pay per token of reasoning, not per month of access.

Track 4 — Real-Time Micro-Commerce: Approved trades settle in USDC on Arc via Circle Wallets. Sub-second finality. No gas volatility. No batching delay. The governed moment the trade earns its execution.

**Why this model requires Arc.**

On Ethereum mainnet, each $0.001 governance payment would cost $2–20 in ETH gas. The entire economic model is structurally impossible on gas chains. Arc's USDC-native settlement with deterministic sub-second finality is the prerequisite — not an optimisation.

**Production track record.**

Kairos has already logged 890+ executed cycles, reached elite trust tier, ranked 4/48 on the ERC-8004 reputation registry, and produced 50+ IPFS-pinned governance artifacts before the hackathon started. This is not a prototype. It is a production system migrated to Arc with the Circle developer platform wired end-to-end.

**Circle products used:** Arc · USDC · Circle Nanopayments · Circle Wallets · Circle Gateway · x402

## CIRCLE PRODUCT FEEDBACK

### Which Circle products did you use?

All five: Arc (settlement layer), USDC (unit of account and gas token), Circle Nanopayments (governance and compute billing), Circle Wallets (agent identity and signing), Circle Gateway (x402 data payment batching).

### Why you chose these products

The core design requirement was an agent that pays for its own governance at sub-cent scale. Circle Nanopayments was the only infrastructure that made this economically viable — batched settlement eliminates per-transaction gas while preserving instant confirmation. Circle Wallets replaced raw EOA key management with MPC-based programmatic signing, which is the correct architecture for an autonomous agent. Arc was non-negotiable: the economic model structurally requires USDC-native settlement with no gas volatility.

### What worked well

**Circle Nanopayments:** The instant EIP-3009 confirmation model was exactly right for governance billing. Having the txHash returned immediately without waiting for Arc block confirmation meant the billing store could update the dashboard in real time. The metadata field was genuinely useful for tagging each payment event with the governance stage name and agent ID.

**Circle Wallets:** MPC-based signing removed the single biggest security risk in autonomous agents — private key exposure. The API is clean and the blockchain support for Arc testnet was straightforward once the wallet set was created.

**x402 via Circle Gateway:** The GatewayEvmScheme pattern (using `extra.verifyingContract` from the 402 response instead of the USDC asset address) was the key discovery. Once we understood this, integration with AIsa's 79-endpoint catalog was seamless. The batched settlement model meant 5 data API calls per cycle settled as a single Arc transaction — exactly the economic efficiency the model needed.

**Arc:** USDC as native gas eliminated the mental overhead of managing two tokens. The testnet faucet was responsive. The block explorer (testnet.arcscan.app) showed transactions appearing reliably within 1-2 seconds.

### What could be improved

**Nanopayments documentation:** The distinction between EIP-3009 instant confirmation and on-chain Arc batch settlement timing is not clearly documented. We spent time debugging what we thought were failed payments before understanding the batching model. A clear diagram showing the confirmation vs settlement lifecycle would help developers significantly.

**Circle Wallets + x402 compatibility:** Circle Wallets uses MPC-based signing which is architecturally incompatible with the BIP-39 mnemonic that `x402_client.mjs` requires for EIP-712 TransferWithAuthorization signing. This forced a two-wallet architecture (Circle Wallets for governance, separate mnemonic wallet for x402 data payments). If Circle Wallets supported EIP-712 typed data signing via the same API surface, agents could use a single identity for all payment types. This would be a significant DX improvement.

**Arc testnet chain ID documentation:** The confirmed chain ID (5042002) was found in the AIsa Arc-x402 repo, not in the primary Arc documentation. The Arc docs should prominently display the chain ID, RPC URL, USDC token address, and Gateway contract address in a single reference table at the top of the getting started guide.

**Nanopayments SDK for TypeScript:** The TypeScript types for `NanopaymentsClient` were incomplete at integration time — several response fields were typed as `any`. Full type coverage would reduce integration errors significantly.

**Circle Gateway deposit discovery:** The requirement to call `deposit()` rather than transferring USDC directly to the Gateway address is not prominently warned against in the docs. We discovered this the hard way. A bold warning at the top of the Gateway integration guide would save developers from losing testnet funds.

### Recommendations

1. Publish a "Kairos-style" reference architecture showing how Circle Wallets + Nanopayments + Gateway + x402 work together in a single autonomous agent system. The agent economy will need this pattern repeatedly.
2. Consider a Circle Wallets SDK method that generates a compatible EIP-712 signer for x402 payments — this would eliminate the two-wallet architecture that currently forces developers to manage both MPC wallets and mnemonic wallets.
3. The AIsa x402 endpoint catalog (79 endpoints, real-time data, $0.0022–$0.12 per call) is one of the most compelling demonstrations of what the agentic economy looks like in practice. Promote it more prominently in Circle developer docs.
4. A Nanopayments "session summary" API — returning total spend, event count, and Arc batch settlement txHashes for a given agent wallet over a time window — would make it much easier to build proof-of-payment dashboards like the /kairos judge view.
